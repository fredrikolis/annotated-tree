Strict-check report — `--strict-check --format json` and MCP `strict_check`

{
  "passed":         bool,          // true iff no annotation AND no rule violations (warnings never flip it)
  "error_count":    integer,       // number of annotation violations
  "files_checked":  integer,       // code files examined
  "annotated_count":integer,       // how many of files_checked already carry a conforming annotation (convergence numerator)
  "violations":     [ Violation, … ],
  "rule_violations":[ RuleViolation, … ],
  "warnings":       [ Warning, … ]  // NON-FATAL advisories; absent (not []) when none
}

Violation = {
  "path":       string,            // path relative to the checked root (unix slashes)
  "line":       integer,           // 1-based line the scan landed on (past shebang/blanks)
  "language":   string,            // resolved language name
  "category":   "missing_annotation" | "malformed_annotation" | "annotation_vacuous",
  "marker":     string,            // the comment delimiter this language expects
  "example":    string,            // a canonical self-conforming line for this language (a guaranteed-valid concrete instance)
  "defect":     {                  // machine-coded delta — the template parts at fault (branch on THIS, not message)
    "missing":  [ "concern" | "non_concern" | "io", … ],   // absent fields;  key omitted when empty
    "vacuous":  [ "concern" | "non_concern" | "io", … ]    // hollow fields;  key omitted when empty
  },
  "expected":   {                  // the canonical contract to converge on (identical on every violation)
    "template":    string,         // "Concern: {what it does} | Non-concern: {what it isn't} | IO: (in) -> out  OR  none"
    "required":    [ string, … ],  // fields that FAIL the check when missing/vacuous (concern, non_concern, io)
    "recommended": [ string, … ]   // advised-only fields (empty: all three are required)
  },
  "found":      string | null,     // the offending line (raw for missing / actual for malformed/vacuous), null if none
  "suggestion": string,            // a FILE-TAILORED candidate: reused text + VACUOUS placeholder slots — must be edited to pass
  "detail":     string             // why it is vacuous (which slot is empty/filler/placeholder); present only for annotation_vacuous
}

RuleViolation = {
  "code":     "denied_dependency" | "dependency_cycle" | "orphan_package" | "unknown_deny_package" | "missing_package_charter",
  "message":  string,              // the architectural `[rules]` finding, human-readable (dual-render)
  "packages": [ string, … ],       // participating package name(s); the ordered node path for a cycle
  "path":     string               // offending package dir, relative to the root (unix slashes); absent when none applies
}

Warning = {
  "code":     "annotation_on_orphan",  // stable dispatch code
  "path":     string,              // the package directory, relative to the checked root (unix slashes)
  "message":  string               // the advisory body, human-readable (dual-render)
}

An agent dispatches on `category`/`code` (stable) and the structured fields, never on
`message`/`detail` prose. On a violation, `defect` is the machine delta (which of the three
fields are missing/vacuous), `expected` is the contract to converge on, and `suggestion` is
a file-tailored stub whose judgment slots are left as `<…>` placeholders — those FAIL
`annotation_vacuous`, so the stub scaffolds the shape but cannot be submitted unedited.
`annotated_count` of `files_checked` is the progress toward full coverage (the TEXT footer
prints "N of M files annotated"). Three violation categories FAIL the check and set a nonzero exit:
`missing_annotation` (no conforming comment), `malformed_annotation` (a comment that is not the
three-field `Concern: … | Non-concern: … | IO: …` shape), and `annotation_vacuous` (the shape is
present but a required field — Concern, Non-concern, or an IO operand — is empty, a filler token, or an
unfilled placeholder — a box-filled stub). Note the deliberate asymmetry: `none` is filler in
Concern/Non-concern but the blessed "no callable contract" value in IO. A directory's `.annotation`
charter breadcrumb (a bare, marker-less three-field line) is enforced by the SAME grammar and, when
malformed/vacuous, appears as a Violation with `language: "charter"` and `path: "<dir>/.annotation"`.
The opt-in `[rules] require_package_charter` gate adds a `missing_package_charter` RuleViolation for a
manifest-bearing package that owns annotated files yet resolves no charter (off by default). `warnings`
are advisory ONLY: they never fail the check, never change the exit code, and are absent
(not `[]`) when empty. One advisory kind today: `annotation_on_orphan` (a package the dependency
graph shows orphaned — imported nowhere and importing nothing internal — that nonetheless carries
annotated files, so its annotations may mislead an agent into treating dead code as live; connects
the annotation and dependency-rule halves of the tool). `annotation_on_orphan` fires only inside an
ecosystem that has real internal structure, so a lone entry-point binary or a single-package repo
never trips it; it is the always-on, ANNOTATED-only counterpart to the opt-in `[rules] forbid_orphans`
`orphan_package` rule (which FATALLY flags every orphan). `path` on a RuleViolation is absent (not null) when no single
directory applies — a cycle spanning many packages, or a deny rule naming an absent package.
